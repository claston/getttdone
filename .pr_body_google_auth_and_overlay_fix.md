## Summary

- Added Google OAuth login/signup flow with backend endpoints (`/auth/google/start`, `/auth/google/callback`) and frontend callback handoff (`auth-callback.html`) that stores session token and redirects safely.
- Extended access-control persistence to support Google identities (`auth_provider`, `provider_user_id`) and one-time OAuth state storage with TTL (`google_oauth_states`) for PKCE/state validation.
- Updated login/signup pages with "Continue with Google" buttons and kept existing email/password auth paths unchanged.
- Fixed session visibility UX in landing/topbar and hardened conversion lock overlay behavior to prevent stale full-screen black overlays from blocking interaction after reload.

## Why

We needed a low-friction social login option (Google) while preserving the current token-based session model and existing email auth. During regression testing, we also found UX inconsistencies: logged-in state not reflected on landing and occasional overlay lock artifacts that could block all screens.

3. Open `http://localhost:3000/login.html` and `signup.html`:
   - Click `Continuar com Google`

   - Verify `localStorage.getItem("gettdone_user_token")` is set.
4. Validate landing session state:
   - Open `index.html` / `ofx-landing.html` logged-in and logged-out
   - Logged-in: show `Minha Conta` CTA and hide `Entrar`.
5. Validate conversion overlay behavior:
   - Open `ofx-convert.html`, reload, navigate between auth pages and back
   - Confirm no persistent black overlay blocks interaction unless quota lock is intentionally opened.
